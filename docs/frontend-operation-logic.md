# Frontend Operation Logic Spec

这份文档不是“当前页面代码做了什么”的流水账，而是后续前端重写时应遵守的交互规范。

目标：

1. 让后续 AI 或人工重写前端时，不把聊天工具最关键的交互逻辑写偏。
2. 让页面行为尽量接近市面上的聊天工具，例如 QQ、微信、企业微信等常见 IM 产品。
3. 在保持当前后端接口协议不变的前提下，定义一套更优雅、更稳定的前端操作逻辑。

---

## 1. 总体交互原则

前端交互应遵循以下原则：

- 聊天优先
  - 用户进入系统后，最重要的是尽快进入“可聊天”的状态。
- 状态连续
  - 会话、消息、未读、历史加载、重连后的恢复都要尽量连续，不让用户觉得“数据断层”。
- 乐观但可回滚
  - 发送消息时允许前端先显示，但必须给失败态和重试手段。
- 与主流聊天工具一致
  - 会话排序、已读、历史分页、失败重试、隐藏会话恢复等行为尽量贴近用户习惯。
- 业务语义优先于 UI 美化
  - 后续 UI 可以重画，但交互语义不能乱。

---

## 2. 最终确定的操作逻辑

下面这些规则视为已经定稿，后续前端重写时默认遵守。

### 2.1 注册与登录

- 注册成功后，自动登录。
- 自动登录成功后，直接进入聊天主界面。
- 不需要用户注册后再手动点一次登录。

设计原因：

- 这更符合现代聊天工具的直觉体验。
- 可以减少一次无意义操作。

### 2.2 会话隐藏与恢复

- 隐藏会话只影响当前用户自己的会话列表展示。
- 隐藏不会删除消息，也不会删除会话本身。
- 以下情况会自动恢复显示该会话：
  - 当前用户再次从好友列表发起会话
  - 对方发来新消息
  - 当前用户再次在该会话中发送新消息

设计原因：

- 这符合聊天软件中“折叠/隐藏入口，不丢失聊天记录”的常见逻辑。

### 2.3 会话列表排序

- 会话列表始终按“最后一条消息时间”倒序排序。
- 最新活跃的会话排在最上面。
- 如果最后消息时间相同，再按后端返回的会话稳定字段兜底排序，避免顺序抖动。

设计原因：

- 这是绝大多数聊天工具的默认行为。

### 2.4 历史消息加载

- 打开会话时，先加载最近一页历史消息。
- 用户向上滚动到聊天框顶部时，继续加载更早消息。
- 历史消息按从旧到新渲染。
- 每次加载更早历史时，页面应保持滚动位置稳定，不能突然跳到底部或顶部。
- 界面需要有明确的“加载中”提示。
- 当没有更多历史时，需要有明确的“没有更多消息”提示。

设计原因：

- 这就是微信、QQ 这一类产品的标准交互方式。

## 3. 推荐采用的完整交互规则

以下规则用于补齐你还没明确拍板的部分。我这里已经按“更合理、更像成熟聊天工具”的方式替你定好了。

### 3.1 登录后的初始化

登录或自动登录成功后，前端应按以下顺序初始化：

1. 保存 token
2. 获取当前用户信息
3. 拉取会话列表、好友列表、申请列表
4. 建立 WebSocket 连接
5. WebSocket 建立成功后，再次刷新侧边栏数据

说明：

- 第二次刷新侧边栏的目的，是让实时连接建立后的在线状态、离线补推、最近消息摘要尽快同步。

### 3.2 默认首页行为

- 登录成功进入聊天主界面后，默认展示会话 tab。
- 如果会话列表非空：
  - 默认不自动打开第一条会话
  - 等用户自己选择
- 如果会话列表为空：
  - 显示空状态，引导用户添加好友或发起会话

设计原因：

- 自动打开第一条会话容易产生误已读和误打断。

### 3.3 点击会话的行为

点击会话后：

1. 高亮当前会话
2. 更新聊天区标题
3. 首次进入该会话时加载最近一页历史消息
4. 历史加载完成后渲染聊天区
5. 若当前会话存在未读消息，则推进已读

补充规则：

- 点击当前已经打开的会话，不重复发起首屏历史加载
- 但应允许用户通过上滑继续加载更早历史

### 3.4 从好友列表发起聊天

点击好友列表里的“发起会话”后：

1. 若该单聊会话不存在，则创建
2. 若已存在但被隐藏，则恢复显示
3. 自动切换到会话 tab
4. 自动进入该会话

这条规则保持和现在后端语义一致。

### 3.5 消息发送逻辑

发送消息时采用“乐观插入 + 状态机”的方式。

前端一条消息应有以下本地状态：

- `sending`
  - 刚点击发送，已经本地显示，但还没确认成功
- `sent`
  - 服务端已确认或回流到本地
- `failed`
  - 发送失败，需要提示用户处理

最终交互规则：

- 用户点击发送后，消息立即出现在聊天区
- 初始状态显示为 `sending`
- 如果服务端回流成功，则转为 `sent`
- 如果发送失败，则转为 `failed`
- `failed` 消息应支持：
  - 手动点击重试
  - 或者右键/长按后重发

设计原因：

- 这是成熟聊天工具更常见、更自然的体验
- 比“发送失败直接消失”优雅得多

### 3.6 WebSocket 断线时的发送策略

采用如下规则：

- WebSocket 断开时，用户仍然可以继续输入消息内容
- 但点击发送后：
  - 消息先进入本地列表
  - 状态标记为 `failed` 或 `waiting`
- 连接恢复后：
  - 不自动静默重发所有失败消息
  - 由用户主动点击重试

为什么这样定：

- 完全禁止输入会让体验很硬
- 自动后台重发所有消息又容易引入重复发送、状态不透明的问题
- 更稳妥的方式是：允许本地保留消息，但最终是否重发交给用户决定

这比“断线时完全不能发”更像成熟产品，也比“后台默默重试”更安全。

### 3.7 收到新消息时的未读规则

未读数只统计“别人发给我、且我当前还没读掉”的消息。

具体规则：

- 自己发的消息不算未读
- 当前未打开的会话收到新消息：
  - 未读数增加
- 当前打开的会话收到新消息：
  - 默认自动已读
  - 不增加未读数

这是你当前系统最适合的默认规则。

### 3.8 当前打开会话的自动已读条件

这里我替你定成：

- 只要会话当前正在打开，并且聊天窗口处于可见状态，就自动推进已读。

不再要求：

- 必须滚动到底部
- 必须额外点击某个位置

设计原因：

- 当前项目还是单页 Web 前端，不需要过度复杂化“阅读确认”的判断
- 对大多数聊天工具来说，只要你正在看这个会话，就会被视作已读

后续如果你想更严格，也可以再升级为：

- 只有窗口在前台 + 已滚到底部 + 新消息进入可视区域才推进已读

但当前阶段没必要先做这么重。

### 3.9 离线补推消息的显示逻辑

离线补推消息应遵循：

- 先进入本地消息缓存
- 再与首屏历史页 merge 去重
- 打开会话前，这批消息仍然算未读
- 打开会话后，统一进入正常已读逻辑

注意：

- 离线补推不是完整历史
- 首次打开会话仍然要补拉最近一页历史

### 3.10 顶部上滑加载的 UX 细节

用户把聊天框滚到顶部附近时：

- 自动触发加载更早历史
- 聊天框顶部显示“加载中”
- 加载成功后隐藏“加载中”
- 若没有更多历史，顶部显示“没有更多消息”
- 这个提示不应该是打断式弹窗，而应是聊天列表内的轻提示

### 3.11 会话摘要更新规则

会话列表中的 `last_message` 更新规则：

- 只要收到或发送了一条新消息，就刷新该会话摘要
- 该会话移动到列表顶部
- 但如果当前收到的是旧离线补推消息，不应错误地把它当成“最新消息”覆盖更晚的新消息

也就是说：

- 前端在更新会话摘要时，要区分“真正的新消息”和“历史补齐/离线回填消息”

### 3.12 在线状态展示

在线状态建议继续展示在：

- 会话列表
- 好友列表

申请列表可以不作为重点展示区，但如果已有数据同步能力，可以保留。

优先级建议是：

1. 会话列表
2. 好友列表
3. 申请列表

### 3.13 好友申请流程

发送好友申请后，前端必须区分后端返回的不同业务结果：

- `pending`
  - 正常发送成功，显示“等待对方处理”
- `already_friends`
  - 提示“你们已经是好友”
- `auto_accepted`
  - 提示“已自动成为好友”
  - 同时刷新会话、好友、申请列表

同意申请后：

- 自动刷新侧边栏数据
- 自动切到会话 tab
- 不强制自动打开会话

拒绝申请后：

- 只刷新申请列表

### 3.14 隐藏会话的交互建议

点击“隐藏会话”后：

- 当前会话从列表移除
- 聊天区回到“请选择会话”
- 不弹确认框也可以
- 但如果后续你想更稳，可以加轻量确认

我建议当前先不加二次确认。

原因：

- 隐藏不是删除
- 风险比删除低
- 二次确认会降低操作效率

### 3.15 退出登录

退出后必须清空：

- token
- 当前用户信息
- 会话列表
- 好友列表
- 申请列表
- 当前会话
- 本地消息缓存
- 历史分页状态
- 已读重试状态

然后回到登录页。

### 3.16 WebSocket 自动重连

采用以下规则：

- 如果 token 仍然有效，则在断线后自动重连
- 重连间隔可固定 3 秒，也可逐步退避
- 重连成功后：
  - 自动刷新会话列表
  - 自动恢复离线补推逻辑

推荐你后续升级成：

- 第 1 次 1 秒
- 第 2 次 3 秒
- 第 3 次 5 秒
- 之后上限 10 秒

这样会比固定 3 秒更优雅。

## 4. 后续前端重写时最容易写错的地方

以下是必须重点提醒 AI 的风险点。

### 4.1 把离线补推误当成完整历史

错误写法：

- 收到离线消息后，就认为这个会话历史已经完整了

正确写法：

- 离线补推只是补一段未读消息
- 首次打开会话仍然要请求最近一页历史

### 4.2 把自己发的消息算成未读

错误写法：

- 当前会话里只要来了消息就加未读

正确写法：

- 未读只统计发给当前用户的消息

### 4.3 上滑加载时不保持滚动位置

错误写法：

- prepend 旧消息后，滚动条直接跳动

正确写法：

- 加载前记录高度
- 加载后修正 `scrollTop`

### 4.4 把隐藏会话写成删除聊天记录

正确语义必须是：

- 只隐藏入口
- 不删消息
- 不删会话

### 4.5 把发送失败静默吞掉

错误写法：

- 用户看到消息已显示，但其实后端没成功
- 页面也不给任何失败提示

正确写法：

- 必须有 `failed` 状态
- 必须允许用户重试

### 4.6 把当前会话收到消息误判为未读

当前推荐逻辑：

- 当前正在看的会话默认自动已读

### 4.7 把历史分页写成一次性全量加载

错误写法：

- 打开会话直接拉全量历史

正确写法：

- 首屏一页
- 上滑继续翻页

## 5. 交给 AI 重写前端时必须附带的硬约束

后续把前端交给 AI 重写时，建议明确写进提示词中的约束如下：

- 不能改变现有后端 API 协议
- 不能改变 WebSocket event type
- 不能把离线补推当成完整历史
- 不能把自己发的消息算作未读
- 不能把隐藏会话实现成删除聊天记录
- 必须支持历史消息分页上滑加载
- 必须支持消息发送失败态与手动重试
- 注册成功后必须自动登录并进入聊天页
- 会话列表必须按最后一条消息时间排序
- 当前打开会话收到新消息时默认自动已读

## 6. 当前这份规范与现有后端的关系

这份文档的设计原则是：

- 尽量不改变后端协议
- 主要优化前端交互语义

其中以下能力已经和当前后端相匹配：

- 注册 / 登录
- 会话打开与恢复显示
- 历史消息分页
- 离线补推
- 已读推进
- WebSocket 消息收发

以下能力如果要真正做得更完整，前端后续还可以继续增强：

- 本地消息发送状态 `sending / sent / failed`
- 失败消息重试按钮
- 顶部“加载中 / 没有更多消息”提示
- 更优雅的断线重连反馈

---

## 7. 一句话总结

后续前端应做成这样：

- 注册成功即进入聊天
- 会话列表始终按最新消息排序
- 打开会话先看最近消息
- 上滑继续加载更早历史
- 当前打开会话默认自动已读
- 消息先本地显示，再根据结果呈现发送成功或失败
- 隐藏会话不等于删除聊天

这套交互逻辑已经足够接近常见聊天工具，也和你当前后端能力基本匹配。
