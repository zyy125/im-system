# IM System Architecture Roadmap

本文档用于把当前项目从“可运行的单体即时通讯系统”逐步演进为“可扩展、可拆分、适合后续做分布式”的架构。

目标不是立刻改成微服务，而是先把当前单体整理成一个边界清晰、职责明确、可测试、可拆分的模块化单体。这样后续无论是继续做单体增强，还是拆成分布式服务，成本都会低很多。

## 1. 当前架构概况

结合当前代码，项目大致分为以下模块：

- `handler`
  负责 HTTP / WebSocket 接口层，处理请求参数、鉴权上下文、响应格式。
- `service`
  负责业务逻辑，如认证、好友、好友申请、会话、消息。
- `repository`
  负责 MySQL / Redis 数据访问。
- `ws`
  负责连接管理、在线状态同步、消息转发、离线消息补推。
- `mq`
  负责 RabbitMQ 生产与消费，目前主要用于消息持久化链路。
- `router` / `app`
  负责依赖装配与启动流程。

当前这套结构已经比“所有逻辑堆在 handler 里”要好，但仍然有几个典型问题：

- `Hub` 责任偏重，同时承担了连接管理、在线状态变更、离线补推、消息路由、在线广播。
- HTTP、WebSocket、MQ 三条链路的协议边界还不够明确，后续扩展时容易互相牵连。
- 测试覆盖不足，后面一旦做架构改动，很难快速确认是否出现回归。
- 当前 Swagger 仅覆盖少量接口，HTTP 协议文档还不完整；WebSocket 与 MQ 事件也缺少独立文档。
- 业务上已经开始形成“会话中心”模型，但后续如果做未读数、最近消息、会话隐藏、群聊，索引模型还需要继续加强。

## 2. 总体演进原则

后续所有改造遵循下面几条原则：

- 先稳单体，再谈分布式。
- 先明确模块边界，再拆物理服务。
- 先补测试和协议文档，再做大重构。
- 先把 `Hub` 变薄，再让消息路由具备跨节点能力。
- 先区分“命令写模型”和“展示读模型”，再扩展复杂功能。

不建议现在直接把项目拆成多个微服务。对当前阶段来说，最优路径是：

1. 把现有单体整理成模块化单体。
2. 给关键业务补测试与协议文档。
3. 抽象消息路由与用户定位能力。
4. 再逐步把部分能力替换为可分布式实现。

### Phase 1：把单体整理成模块化单体

这一阶段的目标是：让代码逻辑先按领域边界拆清楚，即使仍然在一个进程里运行，也已经具备“未来可拆”的结构。

#### TODO

- [ ] 按领域梳理模块边界，至少明确：
  - `auth`
  - `user`
  - `friend`
  - `conversation`
  - `message`
  - `presence`
  - `gateway/ws`
- [ ] 减少跨模块直接操作底层 repo 的情况，尽量通过 service / domain service 协作。
- [ ] 把 handler 中的 DTO、service 中的业务对象、MQ 中的事件对象区分开。
- [ ] 梳理 `app.InitApp` 的依赖注入，把初始化逻辑按模块分段装配，避免一个函数持续膨胀。
- [ ] 逐步减少 `service` 之间过深的相互持有，必要时引入更小粒度接口而不是直接依赖整个 service。

#### 验收标准

- 新增一个业务功能时，能明确知道应该落在哪个模块。
- 大部分改动都局限在单个模块内，不需要跨 5 到 6 个包同时改。
- `app.InitApp` 主要负责装配，不再承担过多业务决策。

### Phase 2：瘦身 Hub，分离连接管理与消息投递

这一阶段是后续分布式演进的关键。

当前 `Hub` 同时做了太多事。后续应把它拆成更清晰的职责层。

#### 目标拆分

- `ConnectionManager`
  只管理连接注册、注销、发送通道。
- `PresenceService`
  负责上下线状态与在线广播。
- `OfflineDeliveryService`
  负责用户上线后的离线消息补推。
- `DeliveryRouter`
  负责“某条消息应该投递给谁、如何投递”。

#### TODO

- [ ] 抽象“用户在哪个节点 / 是否在本节点在线”的能力，例如：
  - `UserLocator`
  - `SessionLocator`
- [ ] 抽象“如何向目标用户投递”的能力，例如：
  - `DeliveryRouter`
  - `LocalDeliveryRouter`
- [ ] 把 `Hub` 中的在线状态写入逻辑下沉到单独服务，不让 `Hub` 直接承担过多数据库 / Redis 语义。
- [ ] 把离线消息加载与补推流程封装成独立流程对象，降低 `Hub.Run` 主循环复杂度。
- [ ] 把在线广播也从 `Hub` 主流程中剥离为单独协作者。
- [ ] 给 `Hub` 增加必要的生命周期说明和时序文档，避免后续自己都难维护。

#### 验收标准

- `Hub` 文件明显变短，主循环只保留连接事件和分发事件。
- 离线补推、在线广播、状态落库不再直接写在同一个 select 分支里。
- 后续如果要支持多节点路由，只需要替换 locator / router 实现，而不是重写整个 `Hub`。

### Phase 3：把 MQ 从“消息搬运工具”升级成“领域事件总线”

当前 RabbitMQ 已经进入链路，但后面不能只把它当成“把消息塞进去再取出来”的工具。

应逐步把 MQ 设计成更稳定的事件分发层。

#### TODO

- [ ] 区分“命令消息”和“领域事件”：
  - 命令：要求系统做某事
  - 事件：表示某事已经发生
- [ ] 定义统一事件结构，例如：
  - `event_id`
  - `event_type`
  - `occurred_at`
  - `trace_id`
  - `payload`
- [ ] 为消息持久化、会话索引更新、通知推送设计分离的消费职责。
- [ ] 增加幂等消费设计，确保消息重复投递不会造成脏数据。
- [ ] 规划死信队列、重试次数、不可恢复错误处理策略。
- [ ] 为关键业务链路考虑 Outbox 模式，避免“数据库成功但 MQ 发送失败”的一致性问题。

#### 验收标准

- MQ 中的消息类型和消费职责清晰，不再混在一起。
- 关键消费者具备幂等性，重复消费不会重复创建数据。
- 后续要加站内通知、推送、风控审计时，可以直接复用事件总线。

### Phase 6：按边界逐步拆服务

只有当前面几阶段完成得比较扎实时，才建议开始物理拆分。

#### 建议拆分顺序

1. `gateway/ws`
2. `presence`
3. `message`
4. `conversation`
5. `friend`
6. `auth/user`

#### 原因

- WebSocket 网关天然适合先独立，因为它最接近连接层。
- Presence 和消息路由通常最先遇到多节点问题。
- 会话、好友、用户资料相对更偏业务数据，可后拆。

#### 验收标准

- 被拆出的服务有独立边界，不是“把一个大泥球切成几个小泥球”。
- 服务之间接口清楚、事件清楚、数据归属清楚。

## 4. 当前最应该立刻做的事

如果只允许做三件事，优先级如下：

1. 补测试
2. 补协议文档
3. 瘦身 `Hub`

原因很简单：

- 没有测试，后面重构很容易把现有功能改坏。
- 没有协议文档，前后端、WebSocket、MQ 的协作边界始终是模糊的。
- `Hub` 不先变薄，后续任何分布式设计都会卡在连接与业务耦合过重的问题上。

## 5. 是否现在就要做的事项

### 测试要不要现在写

要，现在就应该开始。

建议顺序：

1. `service` 单元测试
2. `repository` 集成测试
3. 少量 E2E 黄金链路测试

理由：

- service 测试性价比最高，最能保护你的业务逻辑。
- repository 测试能提前发现索引、唯一约束、事务边界问题。
- E2E 不需要很多，但要覆盖“最关键的几条主流程”。

### Swagger 要不要给每个接口都写

HTTP 接口建议都写。

但要注意：

- Swagger 主要用于 HTTP API。
- WebSocket 协议不要只靠 Swagger，应单独写 `docs/ws-protocol.md`。
- RabbitMQ 事件不要塞进 Swagger，应单独写 `docs/events.md`。

## 6. 简历 / 面试表达建议

这个项目后续写进简历时，不建议只写“做了一个聊天系统”。更好的表达方式是突出你的架构思考和演进能力。

可以重点强调：

- 基于 Go + Gin + GORM + Redis + RabbitMQ 实现即时通讯系统。
- 设计并实现好友申请、会话中心、未读消息、离线补推、在线状态同步。
- 通过会话成员索引维护 `LastReadMsgSeq` / `LastDeliveredMsgSeq`，支持离线补推与未读统计。
- 将系统按 handler / service / repository / ws / mq 分层，逐步演进为可拆分的模块化单体。
- 为后续分布式扩展预留用户定位、消息路由、事件总线等抽象。
